## Concept in my own words

*3–5 sentences. Why does framing exist at all? What invariants does `recv_line()` need to maintain across calls?*
- each connection always has a read and write buffer. So you can shutdown one buffer while the other is open e.g. shutdown write and keep the read open for receiving buffers. 
- You are always interacting with bytes, even if python shows or prints the actual utf8 value, at the lower level it's all just numbers.
- Also even if a connection is terminated between two sockets, the OS uses a TIME_WAIT  value to ensure that there's enough time for the connection to be completely cleaned up before reusing the address.

## Link back

*A3 showed that TCP does not respect message boundaries. A4 imposes them at the application layer with a delimiter (`\n`) plus a persistent buffer that survives across `recv()` calls. The buffer is the key insight: without it, you cannot handle "one line split across two recvs." This pattern (delimiter + buffer) generalizes — B1 will reuse it with `\r\n` for HTTP request lines, and the same `recv_line` becomes the entry point to the HTTP parser.*
