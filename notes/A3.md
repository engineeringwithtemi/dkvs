## Concept in my own words

*3–5 sentences. The phrase to internalize: "TCP is a byte stream, not a message stream."*
- TCP is a byte stream, not a message stream. Meaning it has no concept of message boundaries
- A buffer is a temporary location in memory where data is held temporarily. It's useful for IP communication and async comms between processes. 
- A socket has two buffers. A read buffer where data available for reading is set and a write buffer where data to be written is stored.

## Mastery check

**Q1:** Why does `recv(1024)` sometimes return fewer than 1024 bytes even when the client sent more?
**A:** recv(1024) might return fewer bytes because TCP is byte stream and doesn't guarantee message boundary (messages will be read in the size they're sent. )

**Q2:** Where are the remaining bytes?
**A:** The remaining bytes are the actual bytes of the messages sent by the connection. 

**Q3:** What guarantees does TCP actually provide?
**A:** TCP guarantees that if messages are delivered, they'd be delivered in the right order. 

## Link back

*A2 gave you a connection. A3 reveals the first hard truth about that connection: TCP delivers a stream of bytes, not the messages you sent. One `send(4096)` may arrive as several `recv()` calls; many small `send()`s may coalesce into one `recv()`. The kernel's receive buffer is the intermediate.
